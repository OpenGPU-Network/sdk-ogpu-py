# SDK v0.2.1 — Product Requirements Document

**Status:** Design locked · ready for implementation
**Target version:** `0.2.1`
**Current version:** `0.2.0.15`
**Owner:** Kutay
**Last updated:** 2026-04-11

---

## 1. Overview

`sdk-ogpu-py` v0.2.1 is a structural overhaul of the Python SDK's public API surface. The current SDK exposes ~15–20% of the OpenGPU protocol contract functionality and leaks internal concepts (raw Web3 contract objects, string-based nonce handling, asymmetric read/write paths). v0.2.1 closes most of the coverage gap, introduces a consistent layered architecture, and gives users object-oriented access to on-chain entities (`Source`, `Task`, `Response`, later `Provider` and `Master`).

This is **not** a protocol upgrade. Contract addresses, ABIs, and on-chain semantics are unchanged. All work happens in the Python layer.

---

## 2. Versioning constraint

v0.2.1 bumps the **minor-minor** version only. Per internal convention:

- **0.2.x** — SDK-level changes. Breaking Python API changes are permitted (pre-1.0 semver).
- **0.3.x** — Reserved for protocol-level breaking changes (new contract addresses, ABI shifts, event signature changes). Requires coordinated migration with Provider App, Client dApp, Management dApp, Relay.

Any change requiring a contract redeploy or ABI update is **out of scope** for v0.2.1 and must be scheduled for 0.3.0.

---

## 3. Goals

1. **Close the coverage gap** — expose every function in `NexusAbi`, `ControllerAbi`, `TerminalAbi`, `VaultAbi`, `SourceAbi`, `TaskAbi`, `ResponseAbi` that makes sense from Python (omit admin-only functions, keep user-callable surface).
2. **Introduce a layered architecture** — two clear layers:
   - `ogpu.protocol` — contract-faithful, 1:1 with ABIs, low-level
   - `ogpu.client` / `ogpu.provider` / `ogpu.master` — role-first, opinionated workflows that compose protocol calls
3. **Object-oriented contract instances** — `Source(addr)`, `Task(addr)`, `Response(addr)` as live, stateless proxies that hold only the contract address and issue chain calls on every method.
4. **Symmetric read/write surface** — every contract view has a Python accessor; every user-callable state change has a Python write method.
5. **Consistent signer management** — per-call `signer=` parameter with role-based environment variable fallback.
6. **Eliminate code duplication** — extract the shared transaction/nonce/retry logic from `publish_task`, `publish_source`, `confirm_response`, `set_agent` into a single helper.
7. **Type-safe status codes and enums** — no more raw integers for status, environment, or delivery method.
8. **Honest network I/O surface** — property access is never a chain call; all RPC operations are explicit methods prefixed with `get_`.
9. **Everything on-chain.** The SDK is a contract client. It does not wrap any off-chain indexer, management backend, or REST API. Every `get_*` method on every protocol class hits an OGPU RPC node. This is a hard principle, not a goal — see §4.
10. **Minimal event subscription.** Ship `ogpu.events` with six critical Nexus event watchers (`watch_task_published`, `watch_attempted`, `watch_response_submitted`, `watch_response_status_changed`, `watch_task_status_changed`, `watch_registered`) so users can react to task lifecycle on-chain. Async is confined to this module — the rest of the SDK stays sync.

---

## 4. Non-goals

- **No protocol changes.** No contract redeploy, no ABI updates, no new events, no new Solidity.
- **No async API anywhere outside `ogpu.events`.** The SDK is sync-only in every module EXCEPT `ogpu.events`. Per decision I (§5.9), event subscription requires async and is confined to a single async island inside `ogpu.events`. Users who do not need event streaming never touch async — the rest of the SDK (client, protocol, provider, master, vault) is sync and Jupyter/script-friendly. The `aio` prefix convention is explicitly rejected: the module is `ogpu.events`, not `ogpu.aio.events`.
- **No indexer wrapper. No HTTP dependency. None. Not anywhere in the SDK.** The current `get_confirmed_response` HTTP call to `management-backend.opengpu.network` is removed entirely. There is no `ogpu.indexer` module. There is no fallback, no convenience wrapper, no "fast path" that hits the management backend. The SDK talks to one thing and one thing only: the OGPU RPC node. Users who need aggregate queries across many tasks call the management backend HTTP API directly — it is not the SDK's business. This is a hard non-goal for v0.2.1 **and for every foreseeable version**.
- **`ogpu/service/` is NOT touched. At all.** The `service` submodule is a separate concern — it is the framework that source developers use **inside their Docker containers** to expose functions as network-callable endpoints (via `@expose` decorators, FastAPI server, handlers, logger). It is not part of the blockchain client surface. v0.2.1 focuses exclusively on the blockchain-interaction side of the SDK (`client/`, and the new `protocol/`, `provider/`, `master/`, `events/`, `types/` modules). The existing `ogpu/agent/` directory is **deleted** in Phase 1 — see decision N1 (§5.14). No files under `ogpu/service/` are modified, moved, renamed, or refactored. Its public API is frozen for this release. Think of `ogpu.service` as a different product that happens to live in the same repository.
- **No keystore / mnemonic support.** Raw hex private key and `eth_account.LocalAccount` only. Encrypted keystore deferred.
- **No multicall batching.** Each view call is a separate RPC. Optimization deferred.
- **No CLI.** SDK is library-only.
- **No migration guide document.** Breaking changes are documented in `CHANGELOG.md` and the GitHub release notes. That is all. Users who cannot or will not adapt stay on v0.2.0.15 — PyPI keeps old versions forever, `pip install ogpu==0.2.0.15` always works. The SDK team does not maintain a separate migration guide, shim layer, or deprecation wrapper. Clean break, clean codebase.

## 5. Design decisions (locked in)

### 5.1 — API shape

| ID | Question | Decision |
|---|---|---|
| **A1** | Sync, async, or both? | **Sync-only.** Async deferred to later release when event listening ships. |
| **A2** | Chain reads as property or method? | **Methods.** Property access must never trigger network I/O — debuggers, `__repr__`, and f-strings implicitly invoke property access and would fan out into uncontrolled RPC storms. Methods make network cost explicit via parentheses. |
| **A3** | Keep `get_` prefix on read methods? | **Yes.** Network-I/O libraries in the Python ecosystem (`web3.py`, `boto3`, `google-cloud-*`) all use explicit `get_` prefixes. Drops the Python "property idiom" argument because that idiom applies to local state only. Also mirrors Solidity ABI names (`getSourceParams`, `getRegistrantCount`) and preserves symmetry with `set_` writers. |

### 5.2 — Instance lifecycle

| ID | Question | Decision |
|---|---|---|
| **B1** | `publish_*` return type: string address or instance? | **Clean break.** `publish_source()` returns `Source`, `publish_task()` returns `Task`. No backward-compat shims. Users adapt during 0.2.0.x → 0.2.1 upgrade. `__str__` returns `self.address` for f-string/log ergonomics; `__repr__` returns `<Source 0xabc... @mainnet>` for debugger clarity. `Source == "0xabc..."` comparison returns `False` — accepted break. |
| **B2** | Existence validation on construction? | **Dual API.** `Task(addr)` is lazy — zero RPC at construction, first method call raises `TaskNotFoundError` on revert. `Task.load(addr)` is eager — makes a cheap view call at construction, raises immediately if not on-chain. Both return the same `Task` type; only the validation timing differs. |
| **B3** | Cache or snapshot escape hatch for batch reads? | **`snapshot()` method.** Returns a frozen dataclass with every view field pre-fetched in one logical batch. Instance itself remains stateless — the snapshot is a separate object. Pure no-cache purity is preserved on the instance; performance-minded users opt into snapshotting explicitly. |

### 5.3 — Signer management

| ID | Question | Decision |
|---|---|---|
| **C1** | Where does the signer come from? | **Per-call optional parameter with role-based env var fallback.** Each protocol operation has an associated role (client, provider, master) and reads the corresponding env var (`CLIENT_PRIVATE_KEY`, `PROVIDER_PRIVATE_KEY`, `MASTER_PRIVATE_KEY`) when `signer=` is not passed. If neither is set, raises `MissingSignerError`. |
| **C1-vault** | Vault operations default? | **Explicit signer required.** Vault operations are role-agnostic (client, provider, master all use the same functions). No env var fallback. `signer=` is a required keyword argument. Prevents accidental deposit/withdraw from the wrong account. |
| **C2** | Instance-bound signer (`.connect()` / `.with_signer()`)? | **N/A — not in v0.2.1.** Per-call only. Instance binding may be revisited in a later version if patterns emerge. |
| **C3** | Signer type? | **`str \| LocalAccount`.** Accepts raw hex private key string OR `eth_account.LocalAccount`. LocalAccount support is ~10 lines of code and unlocks forward compatibility with hardware wallet, KMS, and Fireblocks signer classes that implement the Account interface. |

### 5.4 — Return types

| ID | Question | Decision |
|---|---|---|
| **D1** | Navigation return type (e.g. `task.get_source()`)? | **Instance.** Returns `Source`, not `str`. Enables clean chaining: `task.get_source().get_tasks()[0].get_attempters()`. Raw address available via `task.get_source().address`. |
| **D2** | Write method return type? | **`Receipt` dataclass.** Fields: `tx_hash: str`, `block_number: int`, `gas_used: int`, `status: int`, `logs: list`, `timestamp: int`. Unifies all state-changing operations. Not a string (too little info) and not a raw dict (untyped). |
| **D3** | IPFS auto-fetch on `get_params()`? | **Separate method, `get_metadata()`.** `get_params()` is chain-only and returns the raw ABI tuple as a typed dataclass. `get_metadata()` makes an additional IPFS gateway call and returns `dict[str, Any]` — honest about the fact that the SDK does not dictate the shape of off-chain content. Helper builder classes (`SourceMetadata` for source metadata, `TaskInput` for task input payloads) remain available for typed construction on the write path but are optional — plain dicts are always accepted. Note: `TaskInput` keeps its name per L2; there is no `TaskMetadata`. |
| **D3-schema** | Who owns the metadata schema? | **The protocol, not the SDK.** For Source, the Provider App hard-codes the `cpu/nvidia/amd` URL fields; SDK's `SourceMetadata` helper mirrors this requirement but allows arbitrary extras. For Task, only `function_name` is protocol-required (used for routing); `data` and everything else is entirely source-developer-defined. Minimal validation enforces these two invariants; the rest is passed through unchanged. |

### 5.5 — Data sources

| ID | Question | Decision |
|---|---|---|
| **E1** | Should `get_confirmed_response` keep its HTTP path to the management backend? | **No. Removed entirely.** The standalone `ogpu.client.get_confirmed_response(task_address)` function is deleted. Its functionality is replaced by a chain-native convenience method on the `Task` instance class: `Task.get_confirmed_response() -> Response \| None`. Internally it iterates `self.get_responses()` and returns the first one with `is_confirmed() == True`. No HTTP call anywhere. The `ConfirmedResponse` dataclass is also deleted — users get a `Response` instance instead. |
| **E2** | Is an `ogpu.indexer` module (wrapping management-backend REST endpoints) in scope for v0.2.1? | **No. Not in v0.2.1. Not planned for any release.** The SDK is a contract client, not an API client. Users who need indexer-backed aggregate queries call `management-backend.opengpu.network` directly. The SDK's architecture reserves no module for indexer wrapping. This is a principle, not a timeline. |

### 5.6 — Chain and RPC configuration

| ID | Question | Decision |
|---|---|---|
| **F1** | Chain binding scope — global, per-instance, or context manager? | **Global only, with default = MAINNET.** The existing `ChainConfig._current_chain` is kept as a process-wide global. Default flips from `OGPU_TESTNET` to `OGPU_MAINNET` (breaking change for anyone relying on the old default). Per-instance chain override is rejected — v0.2.1 targets single-chain use. Users who want to switch call `ChainConfig.set_chain(ChainId.OGPU_TESTNET)` explicitly. |
| **F2** | Custom RPC endpoint support? | **Yes, via three new public methods on `ChainConfig`.** Users who run their own node (private RPC, Infura-like, local Anvil/Foundry fork) can override the built-in OGPU RPC URL globally. Three new methods: `ChainConfig.set_rpc(url, chain=None)`, `ChainConfig.get_rpc(chain=None)`, `ChainConfig.reset_rpc(chain=None)`. When `chain=None`, operations apply to the current chain. `set_rpc()` internally calls the existing `Web3Manager.update_rpc_url()` and invalidates the cached Web3 instance so the next call reconnects with the new URL. No environment variable fallback in v0.2.1 — single global setter is enough; env var fallback can be added later if Docker Compose users ask for it. |

### 5.7 — Backward compatibility scope

| ID | Question | Decision |
|---|---|---|
| **G1** | Should v0.2.1 include a shim/deprecation layer for v0.2.0.x patterns? | **No. Clean break, no shim, no deprecation wrapper, no migration guide document.** Breaking changes are listed in `CHANGELOG.md` and GitHub release notes. Users who cannot or will not adapt stay on v0.2.0.15 — PyPI preserves old versions. The SDK team does not maintain a parallel compatibility layer. Internal projects (Crunch-Runner, UC Pilot demo, etc.) are not considered blocking users during v0.2.1 development; they will migrate when ready or pin to 0.2.0.x. |
| **G2** | What does "instance return" breaking change look like in practice? | **Exactly as decided in B1.** `publish_source() -> Source`, `publish_task() -> Task`. `__str__(self) -> self.address` is included for f-string/log ergonomics (Python hygiene, not compat). No `__eq__(str)` support. No `isinstance(x, str)` support. Users who receive a `Source` or `Task` from the SDK access `.address` directly when they need the raw string. Every existing example is rewritten in Phase 7 (the final polish/release phase) to use the new pattern. Every new method has a dedicated unit test before release — no code merges without tests. |
| **G3** | Is `ogpu/service/` affected? | **No. Completely out of scope.** See §4 non-goals. Not a single file under `ogpu/service/` is modified in v0.2.1. Its public API is frozen. The service module is the source-developer's Docker container framework — it is a different product living in the same repository. |

### 5.8 — Provider and Master class scope

| ID | Question | Decision |
|---|---|---|
| **H1** | What methods belong on `Provider(addr)` and `Master(addr)`? | **Full composition across Terminal + Vault + Nexus (for Provider), Terminal + Vault (for Master).** See §7 Phase 5 for the complete method list. `Provider` gets read wrappers for all provider-relevant view calls and write wrappers for provider-role state changes (announce_master, set_base_data, set_live_data, register_to, unregister_from, attempt). `Master` gets read wrappers for master-role views and write wrappers for master lifecycle operations (announce_provider, remove_provider, remove_container). Both classes are synthetic — they do not correspond to a single contract ABI but compose calls across Terminal, Vault, and Nexus. |
| **H2** | Where do Vault operations live — module only, `Provider` only, or both? | **(c) Both.** `ogpu.protocol.vault` is the low-level module with every Vault function: `deposit`, `withdraw`, `lock`, `unbond`, `cancel_unbonding`, `claim`, plus all view functions. These are role-agnostic and callable by any account (client, provider, master, arbitrary address). In parallel, `Provider(addr)` exposes convenience wrappers that delegate to the module-level functions with `self.address` filled in as context: `provider.stake(amount, signer=...)` → `vault.lock(amount, signer=...)`, `provider.claim_earnings(signer=...)` → `vault.claim(signer=...)`, `provider.deposit_to_vault(amount, signer=...)` → `vault.deposit(self.address, amount, signer=...)`, `provider.unstake(amount, signer=...)` → `vault.unbond(amount, signer=...)`, `provider.cancel_unbonding(signer=...)` → `vault.cancel_unbonding(signer=...)`, `provider.withdraw_from_vault(amount, signer=...)` → `vault.withdraw(amount, signer=...)`. The convenience wrappers are one-line delegations to the module — zero duplicated logic. Users who do not need the provider context use the module directly; users managing a specific provider use the instance wrappers. The signer is still required on write operations (per C1-vault) — the Provider wrapper does not default the signer to anything; it only saves typing `self.address` as the first argument. |

### 5.9 — Events

| ID | Question | Decision |
|---|---|---|
| **I1** | What is the event subscription API shape in v0.2.1? | **Flat module functions, async generators.** `ogpu.events` exposes one `watch_*` function per critical event. Each is an async generator yielding typed event dataclasses: `async for event in watch_attempted(task_address): ...`. Instance-bound API (`task.events.attempted.stream()`) is deferred to a later release — v0.2.1 keeps the surface small. |
| **I2** | Where does async live? What is the module path? | **`ogpu.events` only. No `aio` prefix.** The events module is the one async island in an otherwise sync SDK. The rest of the SDK (client, protocol, provider, master, vault) stays sync. Users who need event streaming import from `ogpu.events`; users who do not never touch async. No `ogpu.aio.*` namespace exists. |
| **I3** | Is the events module in scope for v0.2.1? | **Yes, minimal scope.** v0.2.1 ships `ogpu.events` with a minimal event surface — six critical events only, flat module API, live streaming only (no historical queries), HTTP filter polling only (no WebSocket provider). |
| **I4** | Which events are in scope for v0.2.1? | **Six critical events from Nexus:** `TaskPublished`, `Attempted`, `ResponseSubmitted`, `ResponseStatusChanged`, `TaskStatusChanged`, `Registered`. These are the events needed for task lifecycle monitoring and provider observation. Terminal and Vault events are deferred. Historical queries (`past_*` functions) are deferred. Instance-bound API (`task.events.*`) is deferred. WebSocket provider (vs HTTP filter polling) is deferred. |

### 5.10 — Exception taxonomy

| ID | Question | Decision |
|---|---|---|
| **J1** | How granular is the exception hierarchy? | **Granular, ~22 concrete exception classes.** All exceptions inherit from a single base `OGPUError`. Concrete classes are grouped by failure domain: resource-not-found (5), state errors (4), permission errors (4), vault errors (4), transaction errors (4), configuration errors (3), IPFS errors (2). Callers can catch the base `OGPUError` for broad handling or any specific subclass for precise handling. No more `raise Exception(...)` anywhere in the touched code. Every exception is raised with rich context in its constructor (e.g. `TaskNotFoundError(address="0xabc")`). |
| **J2** | How are Solidity revert reasons decoded into typed exceptions? | **Pattern-match table with fallback.** The `TxExecutor` in `ogpu/protocol/_base.py` catches `ContractLogicError`, extracts the revert reason string, and looks it up in `REVERT_PATTERN_MAP`. If the reason matches a known pattern (`"NotOwner"`, `"Expired"`, `"InsufficientBalance"`, etc.), the corresponding typed exception is raised. If no match, `TxRevertError(reason=message)` is raised as a generic typed fallback — still better than raw `Exception` because it is typed, contains the reason, and is catchable separately. The pattern map is seeded in Phase 1 from the current contract revert strings (to be verified against the live contracts by Kutay) and is extensible — adding a new pattern is a one-line addition. Solidity 0.8.4+ custom error (`error NotOwner()`) decoding via ABI error types is **deferred to v0.2.2**; v0.2.1 uses string-based pattern matching only. |

### 5.11 — Pagination

| ID | Question | Decision |
|---|---|---|
| **K1** | What does `upper=None` mean on paginated reads? | **Fetch all, with internal chunking.** When `upper=None` (the default), the SDK first calls the matching count function (e.g. `getAttemptCount`), then iterates in fixed-size chunks until all records are retrieved. User gets a single `list[...]` back. Transparent — user does not see the chunking happen. |
| **K2** | Default `lower` value? | **`lower=0`.** Starts at the beginning of the list. Trivially correct. |
| **K3** | Chunk size for internal pagination? | **100.** Constant `_DEFAULT_CHUNK_SIZE = 100` in `ogpu/protocol/_base.py`. Small enough to avoid RPC timeout on any reasonable OGPU node, large enough to keep the call count manageable for typical task sizes. Not user-configurable in v0.2.1 — can be exposed later if someone asks. |
| **K4** | Warning or hard limit when fetching very large lists? | **Neither. Silent.** The SDK does not warn, does not cap, and does not raise when a user requests 100K items. Pagination is the user's choice; the SDK does not second-guess it. No `PaginationLimitError`, no `logger.warning()` noise. Users who want observability wrap the call themselves. |
| **K5** | Return type — `list` or `Iterator`? | **`list[...]`.** v0.2.1 ships list-only returns. Iterator variants (`iter_attempters()`, `iter_tasks()`, etc.) deferred to v0.2.2 if user demand materializes. Most users expect list semantics (len, index, slice, sort) and the internal chunking already handles memory reasonably for typical sizes. |

### 5.12 — Type names and `ogpu.types` structure

| ID | Question | Decision |
|---|---|---|
| **L1** | Rename `ImageMetadata` to something more accurate? | **Yes, rename to `SourceMetadata`.** The current name misleadingly suggests "docker image metadata" but the dataclass actually contains the full off-chain metadata of a Source (cpu/nvidia/amd docker compose URLs, display name, description, logo URL). `SourceMetadata` is semantically correct. Breaking change, documented in CHANGELOG. |
| **L2** | Rename `TaskInput` to `TaskMetadata` for symmetry with `SourceMetadata`? | **No, keep `TaskInput`.** The type is semantically an operational input to a task (function name + data payload + optional extras) — calling it "metadata" would be inaccurate. Method name symmetry (`get_metadata()` on both Source and Task) is sufficient — Python standard library is full of methods whose names don't match their return types (e.g. `dict.keys() -> KeysView`). Type names reflect semantics, not symmetry. |
| **L3** | Single `types.py` file or split into `types/` folder? | **Folder, split by concern.** The types surface has grown too large for a single file (22 concrete exception classes + 5 enums + Receipt + SourceMetadata + TaskInput = 30+ types, ~500+ lines). Split into `ogpu/types/` directory with submodules: `enums.py`, `errors.py`, `receipt.py`, `metadata.py`. `ogpu/types/__init__.py` re-exports everything so user-facing imports stay flat: `from ogpu.types import TaskStatus, OGPUError, Receipt, SourceMetadata, TaskInput`. |
| **L4** | Where do `SourceMetadata` and `TaskInput` live? | **`ogpu/types/metadata.py`.** Consolidated with the rest of the top-level types. The old location `ogpu/client/types.py` is removed; its contents are redistributed across `ogpu/types/` during Phase 1. |

### 5.13 — File structure confirmation

| ID | Question | Decision |
|---|---|---|
| **M1** | Is the `ogpu/protocol/` directory layout in §6.1 final? | **Yes, confirmed.** Layout matches §6.1 architecture diagram: `_base.py` + `_signer.py` for shared infrastructure, one module per singleton contract (`nexus.py`, `controller.py`, `terminal.py`, `vault.py`), one file per instance class (`source.py`, `task.py`, `response.py`, `provider.py`, `master.py`). |
| **M2** | Should `_base.py` be split into separate files (tx executor vs pagination vs errors)? | **No. Keep as single file.** `_base.py` is a private module (underscore prefix) not exposed to users. Consolidating shared utilities (`TxExecutor`, `_paginated_call`, `_DEFAULT_CHUNK_SIZE`, `REVERT_PATTERN_MAP`, `ZERO_ADDRESS`, `load_contract`) in one place keeps the protocol layer navigable. ~300 lines is fine for a private module. |
| **M3** | One file per instance class or single `instances.py`? | **One file per class.** Each class (`Source`, `Task`, `Response`, `Provider`, `Master`) has 15–25 methods and grows to ~200–300 lines. Five separate files are easier to navigate, review, and test. Single `instances.py` would become a 1000+ line monster. |
| **M4** | Where do the module-level contract files live — `ogpu/protocol/` root or nested `_modules/`? | **Protocol root.** `from ogpu.protocol import nexus, controller, terminal, vault` is flat and discoverable. Nested `_modules/` adds unnecessary depth. |
| **M5** | What does `ogpu/protocol/__init__.py` re-export? | **Both instance classes and module-level namespaces:** `from ogpu.protocol import Source, Task, Response, Provider, Master` AND `from ogpu.protocol import nexus, controller, terminal, vault`. Users can access the SDK at either layer through the same package. `__all__` declares all of them for `from ogpu.protocol import *` compatibility. |

### 5.14 — Agent disposition and phase releasability

| ID | Question | Decision |
|---|---|---|
| **N1** | What happens to `ogpu/agent/` in v0.2.1? | **Deleted in Phase 1.** The existing `ogpu/agent/` directory contains only `terminal.py::set_agent()`, which is migrated to `ogpu/protocol/terminal.py::set_agent()`. Once migrated, `ogpu/agent/` has no reason to exist. Agent is a **relationship**, not an entity — a principal (Client or Master) designates any address as their agent via `Terminal.setAgent()`, and that address acts on their behalf by signing transactions with its own private key. There is no "Agent contract" on-chain, no standalone state, and therefore no `Agent` class in the SDK. To act as an agent on behalf of a principal, the user passes the agent's private key as the `signer=` parameter to any write method; the protocol's authorization check (`isAgentOf(principal, agent)`) runs transparently on-chain. **Protocol note:** the `Terminal.AgentSet` event is triggered only by Master or Client (per the contract event trigger table in the Notion protocol docs). **Providers cannot set agents** — they are not principals in the agent-delegation model. Therefore `Master.set_agent(agent, signer=...)` is provided as a convenience wrapper (calls `Terminal.setAgent` with master's signer); `Provider` does NOT get a `set_agent` method. Clients call `terminal.set_agent(agent, True, signer=client_key)` directly since there is no Client instance class. `terminal.set_agent(agent, enabled, signer=...)` is the low-level primitive. `terminal.revoke_agent(agent, signer=...)` is a one-line convenience wrapper calling `set_agent(agent, False, signer=...)`. |
| **N2** | Are phases independently releasable? | **No. Only v0.2.1 as a whole is released.** The §7 preamble is corrected to reflect this. Phases are an internal sequencing of work — each phase has its own verification gate, but none of them produce a release. The only artifact shipped to PyPI is the complete v0.2.1 build, after Phase 7 completes. Phase 1 is explicitly transitional: it keeps `publish_source` and `publish_task` returning `str` temporarily so the refactor doesn't break the in-flight test suite, and Phase 2 flips the return type to `Source`/`Task`. Because phases are not released independently, this temporary `str` return is never observed by end users — they only see the final clean-break API in v0.2.1. |
| **N3** | When does the default chain flip from TESTNET to MAINNET happen? | **Phase 1.** The flip happens as part of Phase 1's `ChainConfig` update. Examples that depend on the testnet default are updated in Phase 1 to prepend `ChainConfig.set_chain(ChainId.OGPU_TESTNET)` so they continue to work. Phase 1 verification requires every testnet-targeting example to have the explicit `set_chain(TESTNET)` call before the refactor lands. |

## 6. Architecture

### 6.1 — Layered module structure

```
ogpu/
├── __init__.py                 # top-level re-exports
│
├── types/                      # enums, dataclasses, exceptions (top-level shared)
│   ├── __init__.py             # re-exports everything — user imports stay flat
│   ├── enums.py                # TaskStatus, SourceStatus, ResponseStatus, Environment, DeliveryMethod
│   ├── errors.py               # OGPUError hierarchy (22 concrete + 7 abstract + base)
│   ├── receipt.py              # Receipt dataclass
│   └── metadata.py             # SourceMetadata (renamed from ImageMetadata), TaskInput, ImageEnvironments
│
├── protocol/                   # LOW-LEVEL — 1:1 with contract ABIs
```

**`ogpu/service/` is explicitly out of scope for v0.2.1.** No file in this directory is modified, moved, renamed, or refactored during any phase of this release. The service module is the framework source developers import inside their Docker containers to expose handler functions — it is conceptually and functionally distinct from the blockchain client (`ogpu.client`, `ogpu.protocol`, etc.). Think of it as a separate product that happens to share a repository. Any cleanup or refactor of `ogpu/service/` is deferred to a future release and is NOT part of v0.2.1's scope, deliverables, tests, or success criteria.

**`ogpu/events/` is the one async island in an otherwise sync SDK.** Per decision I2, the rest of the SDK uses synchronous Web3 calls. Only `ogpu.events` uses `AsyncWeb3` because event subscription is fundamentally a streaming operation that belongs in an async loop. Users who do not need event streaming never import from `ogpu.events` and never touch async. Users who do write one async function and call `asyncio.run()` around it. Per decision I2, there is no `ogpu.aio.*` namespace — `ogpu.events` is the direct path and the ONLY place async lives.

### 6.2 — Layer contract

**Protocol layer** (`ogpu.protocol`):
- Thin wrappers over Web3 contract ABIs
- Function names mirror ABI names (snake_case-converted, `get_` prefix for reads)
- Every Solidity view has a Python function; every user-callable state-changing function has a Python writer
- Returns typed dataclasses, not raw tuples
- Handles signer resolution, nonce management, gas, retry, revert decoding
- No composition, no IPFS fetching, no indexer calls, no opinion

**Role layer** (`ogpu.client`, `ogpu.provider`, `ogpu.master`):
- Composes multiple protocol calls into meaningful workflows
- Example: `client.publish_source(info)` does → IPFS upload → `protocol.nexus.publish_source(params)` → wrap result in `Source` instance
- Adds defaults, validation, convenience
- The surface existing v0.2.0.x users see after migration

**Instance classes** (`Source`, `Task`, `Response`, `Provider`, `Master`):
- Bound to a contract address
- `self.address` is the only persistent state — every method call is a fresh chain read or write
- Navigation methods return other instance types for chaining
- Live in `ogpu.protocol` but re-exported at package level

### 6.3 — Request/response flow examples

**Publish a task (high-level):**
```python
from ogpu.client import publish_task, TaskInfo

task = publish_task(
    TaskInfo(
        source="0xabc...",
        config={"function_name": "predict", "data": {"prompt": "hi"}},
        payment=10**18,
        expiryTime=...,
    )
)
# task is a Task instance with address, ready to query
print(task.address)
print(task.get_status())  # TaskStatus.NEW
```

**Query attempters for a task:**
```python
from ogpu.protocol import Task

task = Task.load("0xabc...")            # eager validation
for attempter in task.get_attempters():  # list[str]
    response = task.get_response_of(attempter)  # Response | None
    if response:
        print(response.get_status(), response.get_data())
```

**Snapshot pattern for dashboards:**
```python
task = Task("0xabc...")
snap = task.snapshot()   # one batch of RPCs → TaskSnapshot (frozen)
print(snap.status, snap.payment, snap.source, snap.winning_provider)
# subsequent accesses are local — no RPC
```

**Vault deposit (explicit signer):**
```python
from ogpu.protocol import vault

receipt = vault.deposit(amount=10**18, signer=my_account)
print(receipt.tx_hash, receipt.gas_used)
```

## 7. Implementation phases

Phases are sequential and **not independently releasable** (per decision N2). Only v0.2.1 as a whole is published to PyPI, and that happens after Phase 7 completes. Each phase has its own verification gate (see §9), but no phase produces a release artifact. Phase 1 in particular is transitional — it temporarily keeps the old API surface intact while introducing the new infrastructure underneath, so the in-flight test suite does not break. End users never see Phase 1's transitional state because the 0.2.0.15 → 0.2.1 jump happens in one atomic PyPI publish at the end of Phase 7.

Phase 1 is **transitional** (per N2) — it keeps the old API signatures unchanged externally while introducing the new internal structure.       REVERT_PATTERN_MAP: dict[str, type[OGPUError]] = {

      ```python
          "SourceInactive":        SourceInactiveError,
      ```

Each snapshot class is a `@dataclass(frozen=True)` that contains every field the corresponding `get_*()` method on the instance class can return, captured in one logical batch by the `snapshot()` method. Instances remain stateless; snapshots are separate immutable objects (per B3).

```python
@dataclass(frozen=True)
class SourceSnapshot:
    address: str
    client: str
    status: SourceStatus
    params: SourceParams
    task_count: int
    registrant_count: int
    # Note: get_tasks(), get_registrants(), get_metadata() are NOT included in the snapshot
    # because they are paginated or IPFS-fetching calls; users request them explicitly.

@dataclass(frozen=True)
class TaskSnapshot:
    address: str
    source: str                  # raw address; user can wrap in Source() if needed
    status: TaskStatus
    params: TaskParams
    payment: int
    expiry_time: int
    delivery_method: DeliveryMethod
    attempt_count: int
    winning_provider: str | None
    # attempters, responses, metadata NOT included — paginated/IPFS.

@dataclass(frozen=True)
class ResponseSnapshot:
    address: str
    task: str                    # raw address
    params: ResponseParams
    data: str
    status: ResponseStatus
    timestamp: int
    confirmed: bool
```

`Source.snapshot()`, `Task.snapshot()`, `Response.snapshot()` each issue the minimum number of RPC calls needed to populate every field (one call per view, no pagination). Coverage rationale: snapshot is for "capture this state right now" — pagination-heavy or IPFS fields are still available via the regular `get_*()` methods but the user pays that cost explicitly.

**Verification:**
- Unit tests with mocked Web3 contract for each read method (eth-tester or monkeypatched `functions`)
- Integration test on testnet: `publish_task → Task instance → navigate to source → list attempters → fetch response → confirm`
- Snapshot test: `task.snapshot()` returns a populated frozen dataclass, RPC call count is bounded

### Phase 6 — Events (the async island)

**Goal:** Ship the minimal `ogpu.events` module with six critical event watchers. This is the ONE async module in the SDK — everything else stays sync.

**Scope (minimal per decisions I3, I4):**
- Six critical events only: `TaskPublished`, `Attempted`, `ResponseSubmitted`, `ResponseStatusChanged`, `TaskStatusChanged`, `Registered` (all from Nexus)

- Flat module API — no instance-bound `task.events.*` API

## 8. Testing strategy

**Hard rule:** No function ships without a unit test. No release happens without a green test run. This is not a best-effort target — it is a release gate. Per decision G2, the SDK team will not release v0.2.1 until every public function, method, and class behavior has a passing unit test and the integration test suite has been exercised end-to-end on testnet at least once by the author before tagging the release.

### 8.1 — Unit tests

**Scope:** Every public function and every public method on every instance class in `ogpu/protocol/`, `ogpu/client/`, `ogpu/provider/`, `ogpu/master/`, `ogpu/events/`, and `ogpu/types/`. Isolated from chain via mocks.

**Non-negotiable rules:**
- Every new function in v0.2.1 ships with at least one unit test — no exceptions.
- Every existing function that is refactored (e.g. `publish_task` migrating to `TxExecutor`) retains its existing test coverage AND gains new tests for the refactored code path.
- A pull request that adds a method without a corresponding test is not merged.
- The test suite runs on every CI build and blocks merge on any failure.
- Coverage target: **≥90% lines** in `ogpu/protocol/`, `ogpu/types/`, `ogpu/client/`, `ogpu/provider/`, `ogpu/master/`, `ogpu/events/`. Anything under `ogpu/service/` is ignored by coverage (it is out of scope, see §4).
- Async tests in `ogpu/events/` use `pytest-asyncio`. New dev dependency.

**Approach:**
- Mock `web3.contract.Contract` via `unittest.mock` or a small stub for sync modules
- Mock `AsyncContract` and `AsyncFilter` for `ogpu/events/` tests — synthetic log lists fed into filter `get_new_entries()`
- Use `eth-tester` for light integration cases where mocking is too verbose
- Each test asserts:
  - Correct contract method is called with correct arguments
  - Return type matches spec
  - Exceptions raised for invalid inputs / revert conditions
  - Signer resolution path (env var vs explicit parameter vs LocalAccount)

**Key test cases:**
- `Task.load("0x000...0")` raises `TaskNotFoundError`
- `Task("0xabc...").get_status()` returns `TaskStatus` enum, not raw int
- `task.get_source()` returns a `Source` instance (not `str`), with correct address
- `task.snapshot()` makes exactly N RPC calls (where N is the number of view methods), no more
- `Receipt` fields are populated from `TransactionReceipt` correctly
- `resolve_signer(None, role=Role.CLIENT)` reads `CLIENT_PRIVATE_KEY`, raises if unset
- `resolve_signer(None, role=None)` (vault) raises `MissingSignerError` regardless of env
- `resolve_signer(LocalAccount instance, ...)` returns it unchanged
- Old nonce retry test cases pass against the new `TxExecutor`

### 8.2 — Integration tests

**Scope:** Full SDK ↔ testnet round-trips.

**⚠️ REPLACED by notebook scenarios — see decision O2 (§13).** The originally planned `tests/integration/` pytest suite was dropped in favor of notebook-based scenarios under `examples/scenarios/`. Rationale: integration scenarios are already manual (funded wallet required, rate limits, unbonding waits) and are more valuable as teaching/reference material than as a CI gate. The 7 end-to-end scenarios below map 1:1 to the notebook filenames; walkthrough is by Kutay, not Claude Code (see feedback memory `no_integration_tests_without_ask.md`).

**Original approach (superseded):**
- ~~Dedicated `tests/integration/` directory, marked with `@pytest.mark.integration`~~
- ~~Run against OGPU testnet with a funded test wallet (key in `.env.test`, git-ignored)~~
- ~~CI runs integration suite on main branch pushes~~

**Current approach:**
- `examples/scenarios/*.ipynb` — one notebook per scenario, markdown narrative + code cells + committed last-run outputs
- Kutay walks through each notebook manually on testnet; outputs are committed back so readers see real addresses/status/receipts
- No pytest suite, no CI integration gate, no `.env.test` convention

**End-to-end scenarios:**

1. **Client happy path**
   - `publish_source` → assert returned `Source.get_status() == SourceStatus.ACTIVE`
   - `publish_task` → assert `Task.get_status() == TaskStatus.NEW`
   - Wait for attempt
   - Fetch responses, confirm response
   - Assert `Task.get_status() == TaskStatus.FINALIZED`

2. **Task cancellation**
   - Publish task
   - Cancel before any attempter
   - Assert `TaskStatus.CANCELED`
   - Assert cancellation after attempt raises `TxRevertError`

3. **Source update**
   - Publish source with `minPayment=X`
   - `source.set_params(new_params_with_Y)`
   - Assert `source.get_params().minPayment == Y`

4. **Source inactivation**
   - Publish source
   - `source.inactivate()`
   - Assert `SourceStatus.INACTIVE`
   - Assert new `publish_task` to this source raises

5. **Vault lifecycle**
   - `deposit(amount)` → assert `balance_of` increases
   - `lock(amount)` → assert `lockup_of` increases, `balance_of` decreases
   - `unbond(amount)` → assert `unbonding_of` populated
   - Wait unbonding period
   - `claim()` → assert balance available

6. **Master-provider pairing**
   - `master.announce_provider(provider, amount)` → `Terminal.is_provider(provider) == True`
   - `provider.announce_master(master)` → `Terminal.master_of(provider) == master`
   - `terminal.provider_of(master)` returns provider

7. **Provider task execution** (requires a source running on testnet or a stub source)
   - Provider registers to a source
   - Client publishes task to that source
   - Provider calls `attempt(task, payment)`
   - Provider calls `submit_response(params)`
   - Client sees response via `task.get_responses()`

8. **Snapshot consistency**
   - `task.snapshot()` returns a frozen dataclass
   - Subsequent `task.get_status()` returns potentially different value (confirms snapshot is frozen)

- Add `pyright` as secondary type checker

### 8.4 — Lint / format

- `ruff check` for style
- `ruff format` for consistency
- Both run in CI

### 8.5 — No backward compatibility test

Per decision G1, v0.2.1 is a clean break. There is no compatibility shim layer, no deprecation wrapper, and no `tests/compat/` suite. v0.2.0.15 behavior is not preserved. Users who want the old behavior install the old version from PyPI. The SDK team does not write tests for behaviors that were intentionally removed.

### 8.6 — Service module exclusion

`ogpu/service/` is out of scope for v0.2.1 (see §4). Its existing tests are left untouched. No new tests are written for service code. Coverage reports exclude `ogpu/service/` from the target.

## 10. Risks and mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Existing 0.2.0.x users get upset by breaking changes | Medium | Medium | Clean, honest CHANGELOG with every breaking change listed; clear error messages; prominent version warning in README; users who cannot adapt pin to `ogpu==0.2.0.15` on PyPI. Per G1, no migration guide document is produced. |
| Protocol team changes a contract while 0.2.1 is in flight | Low | High | Coordinate with protocol team before starting Phase 1; lock ABI versions in SDK; if a contract change lands, defer the affected phase to 0.3.0 |
| TxExecutor refactor introduces a regression in nonce handling | Medium | High | Replay all existing nonce-related test cases against the new code before deleting the old implementation |
| Vault integration tests consume real testnet OGPU | Low | Low | Use a dedicated test wallet with a small funded balance; monitor balance in CI |
| Instance class method count explodes and becomes hard to maintain | Medium | Medium | Keep strict layering — all shared logic in `_base.py`; code review focused on duplication |
| IPFS gateway instability breaks `get_metadata()` | Medium | Low | Configurable gateway URL via env var; retries; clear error message when all gateways fail |
| Phase 1 transitional state accidentally lands on PyPI | Low | High | Enforce N2: no PyPI push before Phase 7 completes. CI job blocks `pyproject.toml` version bump commits outside the Phase 7 branch. |
| `ogpu/agent/` deletion breaks internal tools that import from `ogpu.agent` | Low | Low | Grep the repo for `from ogpu.agent` imports in Phase 1, update them all; grep external OpenGPU projects (Crunch-Runner, Relay, Management BE) — none of those currently import from `ogpu.agent`, verified by Kutay before Phase 1 starts. |

## 12. Success criteria

v0.2.1 is considered successful when:

1. **ABI coverage** — Every user-callable function in `NexusAbi`, `ControllerAbi`, `TerminalAbi`, `VaultAbi`, `SourceAbi`, `TaskAbi`, `ResponseAbi` has a Python accessor.
2. **Instance classes** — `Source`, `Task`, `Response`, `Provider`, `Master` are all usable end-to-end and pass their integration tests.
3. **Zero code duplication** — The `TxExecutor` pattern replaces every copy-pasted retry/nonce/gas block; repository line count for tx-sending code drops by ≥200 lines.
4. **Type safety** — `mypy --strict` passes on `ogpu/protocol/`, `ogpu/types/`, `ogpu/client/`, `ogpu/provider/`, `ogpu/master/`, `ogpu/events/`.
5. **Test coverage** — Every new function and method has at least one unit test. Line coverage ≥90% on touched files. Full test suite runs green before PyPI push. (See §8 and §9 for the release gate.)
6. **No service module changes** — `ogpu/service/` diff is empty across the entire release. Not a single file modified.
7. **Events work end-to-end** — `ogpu.events.watch_attempted` yields a real event within 5 seconds of an `Nexus.Attempted` emission on testnet. All six `watch_*` functions pass their integration tests.
8. **Async isolation** — The only async code in v0.2.1 lives under `ogpu/events/`. Verified by grep for `async def` and `await` in other modules — expected count: zero outside `ogpu/events/`.
9. **Published to PyPI** — `pip install ogpu==0.2.1` works, all examples under `examples/` run successfully from a fresh venv.
10. **CHANGELOG is complete** — Every breaking change is listed. No separate migration guide exists (per G1). CHANGELOG and GitHub release notes are the only documentation of changes.
11. **Chain-only** — Zero HTTP calls remain in the SDK for indexer/management-backend operations. The only network protocols the SDK speaks are JSON-RPC to an OGPU node (sync + async) and IPFS gateway HTTP for metadata resolution. No management-backend, no indexer, no REST API wrapping. Verified by grep for `management-backend` and `requests.get` in the touched modules.
